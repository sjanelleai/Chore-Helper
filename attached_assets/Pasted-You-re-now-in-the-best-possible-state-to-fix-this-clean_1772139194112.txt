The “Once and for all” fix (frontend contract)

Principle

The chore list UI must be a pure function of:
	1.	chore_catalog (what chores exist + points)
	2.	daily_status_v2 rows for (child_id, date_key) (what’s done/pending today)

No local booleans, no “toggle state” stored in component state, no reliance on mutation return as truth.

⸻

Step 1 — Lock in ONE canonical dateKey

Make a single utility used everywhere:

export function localDateKey(d = new Date()) {
  const y = d.getFullYear();
  const m = String(d.getMonth() + 1).padStart(2, "0");
  const day = String(d.getDate()).padStart(2, "0");
  return `${y}-${m}-${day}`; // local YYYY-MM-DD
}

Do not use toISOString().slice(0,10) anywhere.

Use this same dateKey for:
	•	the status query
	•	the toggle RPC argument
	•	progress counters
	•	daily email summary logic (if it’s computed in frontend)

⸻

Step 2 — Make useChores() return status-driven items

This must be the core mapping:

const dateKey = localDateKey();
const statusRows = await supabase
  .from("daily_status_v2")
  .select("chore_id,status")
  .eq("child_id", childId)
  .eq("date_key", dateKey);

const statusById = new Map(statusRows.data?.map(r => [r.chore_id, r.status]) ?? []);

const chores = catalog.map(chore => {
  const status = statusById.get(chore.id) ?? "unchecked";
  return {
    ...chore,
    status,
    isApproved: status === "approved",
    isPending: status === "pending",
  };
});

Important: do not derive status from points_ledger. Ledger is balance/earned, not “done list”.

⸻

Step 3 — Sort by status every render (stable)

const rank = { unchecked: 0, pending: 1, approved: 2 } as const;

chores.sort((a, b) =>
  (rank[a.status] - rank[b.status]) ||
  a.title.localeCompare(b.title) ||
  a.id.localeCompare(b.id)
);

This guarantees “checked off goes to bottom” always.

⸻

Step 4 — After toggle, force the ONE query to refresh

No guesswork. After toggle_chore succeeds:
	•	update cache optimistically (instant UX)
	•	then invalidate the exact chores query key

Example:

const choresKey = ["chores", familyId, childId, dateKey];

onSuccess: (resp) => {
  queryClient.setQueryData(choresKey, (old) => {
    if (!old) return old;
    return old.map(c =>
      c.id === choreId ? { ...c, status: resp.status ?? (resp.completed ? "approved" : "unchecked") } : c
    );
  });

  queryClient.invalidateQueries({ queryKey: choresKey });
}

Do not use unstable keys (catalog length, etc.).
Do not rely on prefix invalidation only.

⸻

Step 5 — Add one “truth check” log (dev only)

For future debugging, add a dev-only log when a chore is toggled:
	•	log childId, choreId, dateKey
	•	then after refetch, log the returned statusRows count

This catches dateKey mismatches and silent policy errors in seconds.

The likely reason it still isn’t working right now
