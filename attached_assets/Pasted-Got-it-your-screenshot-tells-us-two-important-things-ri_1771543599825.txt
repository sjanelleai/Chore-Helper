What your developer should review (Supabase QA checklist)

A) First: identify where “family members” are stored

From your screenshot, family_config does not store family members by itself unless you’re putting them inside value JSON.

So your dev needs to confirm which one is true:

Option 1 (likely): You have tables like families / family_members / children

If so, “Add family member” must insert into those tables (not shown in your screenshot).

Option 2: Members are stored inside family_config.value

Example:

{
  "children": [
    { "id": "child_1", "name": "Fitz" },
    { "id": "child_2", "name": "Merckx" }
  ]
}

If this is your design, then “Add family member” must do an UPDATE to JSONB (or an UPSERT) — not an INSERT into some other table.

✅ Your dev needs to answer this first because the required queries differ.

⸻

B) Expected Supabase behavior when adding a family member (what should happen)

If you store children inside family_config.value (simple beta approach)

Expected sequence:
	1.	Read the family config row

	•	SELECT * FROM family_config WHERE key = <family_key>

	2.	If config row doesn’t exist, create it

	•	INSERT family_config (key, value) ...
OR better: UPSERT on key to avoid “already exists” failures.

	3.	Update JSON to append the new child

	•	UPDATE family_config SET value = <merged json> WHERE key = <family_key>

The #1 failure here:
	•	Insert tries to create a duplicate key → violates unique constraint → UI silently shows nothing.

So in QA the dev should confirm they are using upsert (or handling “already exists” explicitly).

⸻

If you store children in a family_members table (more scalable)

Expected sequence:
	1.	Ensure the parent has/creates a family row
	2.	Insert the child into family_members (or children)
	3.	Optionally ensure family_config exists for that family

The #1 failure here:
	•	RLS blocks insert OR FK fails because the “family” row doesn’t exist.

⸻

C) The #1 thing to check when “nothing happens”: the actual error object + Supabase logs

Tell your dev: during QA, capture and paste the full Supabase error object from the failed action:
	•	error.code
	•	error.message
	•	error.details
	•	error.hint

And in Supabase Dashboard check:
	•	Authentication → Logs (for auth-related blocks)
	•	Database logs (for constraint / RLS issues)

Typical smoking-gun messages:
	•	new row violates row-level security policy
	•	duplicate key value violates unique constraint "family_config_key_key"
	•	violates foreign key constraint
	•	null value in column ... violates not-null constraint

If you see one of those, the fix is immediate.

One specific issue visible in your screenshot: email_send_log schema mismatch

You’ve got both:
	•	family_id uuid
	•	family_key text

That’s fine only if your code is consistent. But if your app/runner uses family_key and your DB constraints/logging still use family_id, you’ll get weird behavior.

Have your dev confirm:
	•	What is the PRIMARY KEY on email_send_log right now?
	•	Is it (family_id, date_key) or (family_key, date_key)?

If the primary key is still (family_id, date_key) and you’re writing by family_key, inserts can fail or you’ll get duplicate prevention that doesn’t work.

(You don’t need to fix this to add a family member, but it will matter for nightly sends.)
