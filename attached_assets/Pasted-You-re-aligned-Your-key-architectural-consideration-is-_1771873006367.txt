You’re aligned. Your “key architectural consideration” is the real one: kids won’t have a Supabase Auth session, so you can’t rely on RLS + the Supabase JS client for kid reads/writes. The clean approach is exactly what you wrote: kid-only reads/writes go through RPCs (or Edge Functions) that validate a kid_token against a kid_sessions table.

Below is a build plan you can hand to your frontend dev. It’s scoped to app changes only, with SQL placeholders for your DB dev to implement in v7.

EPIC Build Plan: Parent Auth + Kid Join + Parent Portal PIN

Guiding decisions
	•	Parents use Supabase Auth (email/password) → existing RLS-based Supabase JS reads/writes remain valid for parents.
	•	Kids use family number + kid PIN → no Supabase Auth session for kids.
	•	Kids use kid_token (Pattern A) → token stored in kid_sessions with expiry.
	•	Frontend reuses existing pages; UI adapts by mode (parent vs kid) rather than separate /kid routes.

⸻

Task 1 — Add “mode” to app auth state (Parent / Kid / Guest)

Deliverable

A single source of truth for session mode:
	•	parentSession: Supabase Auth session (existing)
	•	kidSession: stored in sessionStorage (new)
	•	mode = "parent" | "kid" | "guest"

Frontend implementation

Create useAppSession() (or extend auth context):

type KidSession = {
  familyId: string;
  familyNumber: string;
  kidToken: string;
  childId?: string;
  childName?: string;
  expiresAt: number; // unix ms
};

function loadKidSession(): KidSession | null {
  const raw = sessionStorage.getItem("kidSession");
  if (!raw) return null;
  try {
    const s = JSON.parse(raw) as KidSession;
    if (!s.expiresAt || Date.now() > s.expiresAt) return null;
    return s;
  } catch {
    return null;
  }
}

function saveKidSession(s: KidSession) {
  sessionStorage.setItem("kidSession", JSON.stringify(s));
}

Mode resolution:
	•	If parent auth session exists → mode=parent
	•	Else if kidSession exists and not expired → mode=kid
	•	Else → mode=guest

Task 3 — Centralize data access behind a “data adapter” layer

Goal

Keep existing parent Supabase queries intact, but for kids route data through token-backed endpoints.

Deliverable

A “data adapter” that exposes the same interface for both modes:
	•	getChores()
	•	toggleChore()
	•	getRewards()
	•	redeemReward()
	•	getPointsSummary()
	•	getBadges() (if needed)
	•	getDailySummary() (optional)

Parent mode implementation

Keep current Supabase JS client + RLS reads/writes.

Kid mode implementation

Replace Supabase JS reads/writes with RPC calls that accept kid_token:
	•	[SQL PLACEHOLDER] kid_get_catalog(p_kid_token) (chores + rewards)
	•	[SQL PLACEHOLDER] kid_get_points(p_kid_token, p_child_id)
	•	[SQL PLACEHOLDER] kid_toggle_chore(p_kid_token, p_child_id, p_chore_id, p_date_key)
	•	[SQL PLACEHOLDER] kid_redeem_reward(p_kid_token, p_child_id, p_reward_id)

Frontend pattern:

if (mode === "parent") return supabase.from(...).select(...)
if (mode === "kid") return supabase.rpc("kid_get_catalog", { p_kid_token: kidToken })

Important: For kids, you will mostly use .rpc(...) only.
