Here are the only adjustments I’d make before you hand it to your dev, so it’s tighter and less likely to introduce new behavior.

Adjustments to T001 (Bug 1)

1) Don’t add a “pending” toast to parent toggle

In your system, parent toggle should never yield pending. Parent actions are either:
	•	create approved immediately, or
	•	remove the row (uncheck)

So a “pending” toast in parent mode is either dead code or masks a real bug. Keep “pending” toast for kid toggle only.

Replace that line with:
	•	Parent toggle toast should reflect the returned status:
	•	approved → “+X points earned”
	•	unchecked → “-X points removed” (if you show it)
	•	error → show error

2) Fix the real root cause: remove catalog?.length from query keys

This is the biggest improvement and it’s deterministic.

Using catalog?.length in the key creates a new “identity” for the chores query every time catalog changes, and can cause:
	•	invalidation missing the active key
	•	refetch blocked by enabled momentarily
	•	flicker / stale list that looks “not checked”

Change query keys from:
	•	["chores", activeChildId, catalog?.length]

To:
	•	["chores", familyId, activeChildId, dateKey] (or similar)
	•	and keep catalog as a dependency via enabled only, not in the key.

This one change often makes the visual reorder bug disappear without adding refetch hacks.

3) Prefer queryClient.invalidateQueries({ queryKey: [...] }) with the full key

Instead of invalidateQueries({ queryKey: ["chores"] }), do:
	•	invalidate the exact key you use (chores + points)

If your key includes dateKey, include it.

4) Stable sort is good—but make sure the key is stable too

Add the tie breaker (id/title). That’s correct.

Adjustments to T002 (Bug 2)

You nailed it. The only additional detail to add:

1) Use inputMode="numeric" + pattern="[0-9]*" (mobile UX)

This avoids the iOS keyboard being annoying and reduces bad input.

2) Parse on blur OR save (either is fine)

String state is the right approach. I’d parse on save only.
