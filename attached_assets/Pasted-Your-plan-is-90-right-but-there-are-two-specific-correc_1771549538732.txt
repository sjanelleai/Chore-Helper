Your plan is 90% right, but there are two specific corrections you need to make so you don’t accidentally bake in the wrong architecture again.

✅ What’s right (keep)
	•	Frontend refactor to match the audited schema is the correct primary move.
	•	Don’t touch DB unless blocked is correct.
	•	ensure_family_exists RPC (or equivalent) is the right “backend primitive” to create families + family_members + family_settings for new users.
	•	Updating all references:

	•	settings via family_settings

❌ The two fixes you must make

1) Stop using family_config for chores/rewards “catalog” (for now)

You can store catalog JSON in family_config, but it’s not part of the audited contract you’re trying to conform to, and it’s a trap during refactor because it adds another moving part.

Better for this refactor:
	•	Keep catalog in frontend constants for the moment (or wherever it already lives)
	•	Only re-introduce catalog persistence after core auth/family/child flows are stable

If you insist on catalog persistence, treat it as Phase 2, not part of “make the app work.”

2) Do not remove PIN/avatars just because “columns aren’t in the audit”

The audit you pasted is a snapshot of key tables, not proof that those columns never existed or aren’t needed.

Correct approach:
	•	Remove PIN/avatars only if:
	•	the actual children table truly lacks those columns (confirmed in Supabase table editor), and
	•	you’ve decided the feature is out of scope for this build

Otherwise you risk deleting UX that you actually want.
